## R-001: GitHub Actions runner for Xcode 15+

**Decision**: Use `macos-15` runner image (GitHub-hosted), which ships with Xcode 16.x.
Xcode 15+ is also available on `macos-14` (M1 runners). Use `macos-15` for consistency
with Xcode 16 availability, but pin Xcode version explicitly via `xcode-select` to avoid
drift when Apple updates the runner image.

**Rationale**: GitHub-hosted macOS runners include the full Xcode toolchain. `macos-15`
is the current generation; M1 (`macos-14`) is cheaper per-minute but both work identically
for command-line builds. Pinning Xcode version prevents silent breakage when GitHub updates
the runner image.

**Build invocation**:
```bash
xcodebuild \
  -project Drawl.xcodeproj \
  -scheme Drawl \
  -configuration Release \
  -derivedDataPath build \
  -destination 'platform=macOS,arch=arm64' \
  clean build
```

**Test invocation**:
```bash
xcodebuild test \
  -project Drawl.xcodeproj \
  -scheme DrawlTests \
  -destination 'platform=macOS,arch=arm64' \
  -resultBundlePath build/TestResults.xcresult
```

**Alternatives considered**: Self-hosted runner (rejected — requires maintainer to operate a Mac builder machine, too much ops overhead for a small open source project).

## R-002: whisper.spm branch dependency in CI

**Decision**: Accept branch-pinned dependency as-is for now (`branch: master`). GitHub
Actions resolves SPM packages on each CI run from source, so it is not strictly
reproducible but is adequate for a small project. Add a `Package.resolved` file to the
repository (generated by `xcodebuild -resolvePackageDependencies`) and commit it, so
CI uses the pinned revision rather than HEAD.

**Rationale**: Committing `Package.resolved` freezes dependency versions reproducibly
without switching from a branch to a tagged release (which whisper.spm does not yet publish).
If the upstream branch breaks, CI will catch it; the maintainer can update the resolved file
at that point.

**Implementation note**: Run `xcodebuild -resolvePackageDependencies` locally and commit
the resulting `Drawl.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved`.
Add this path explicitly to `.gitignore` exclusions (currently only `xcuserdata/` is ignored).

**Alternatives considered**: Mirror the dependency (rejected — maintenance overhead).
Switch to a tagged release (preferred long-term, but not yet available upstream).

## R-003: Code-signing a macOS Developer ID app in headless CI

**Decision**: Import the Developer ID certificate and private key from a GitHub Secret
(base64-encoded .p12) into a temporary macOS keychain for each CI run. Use `codesign
--deep --force --verify --options runtime` with the Developer ID Application identity.
Clean up the temporary keychain after signing.

**Key steps**:
```bash
# Create a temporary keychain
security create-keychain -p "$KEYCHAIN_PASSWORD" build.keychain
security default-keychain -s build.keychain
security unlock-keychain -p "$KEYCHAIN_PASSWORD" build.keychain

# Import the certificate
echo "$CERTIFICATE_P12_BASE64" | base64 --decode > certificate.p12
security import certificate.p12 \
  -k build.keychain \
  -P "$CERTIFICATE_P12_PASSWORD" \
  -T /usr/bin/codesign

# Allow codesign to access the key
security set-key-partition-list \
  -S apple-tool:,apple:,codesign: \
  -s -k "$KEYCHAIN_PASSWORD" build.keychain

# Sign the app
codesign \
  --deep --force --verify --verbose \
  --options runtime \
  --sign "Developer ID Application: [IDENTITY]" \
  build/Build/Products/Release/Drawl.app
```

**Required GitHub Secrets**:
- `CERTIFICATE_P12_BASE64` — Developer ID Application certificate + private key as base64-encoded .p12
- `CERTIFICATE_P12_PASSWORD` — passphrase for the .p12
- `KEYCHAIN_PASSWORD` — arbitrary password for the temporary keychain (can be `${{ secrets.GITHUB_TOKEN }}` or a dedicated secret)

**Rationale**: Temporary keychain is the standard approach recommended by Apple and GitHub
documentation. It avoids polluting the runner's login keychain and is automatically
destroyed at the end of the job step.

**Alternatives considered**: Fastlane `match` (rejected — adds dependency on Fastlane Ruby toolchain; overkill for a single-certificate project). macOS Runner's existing keychain (rejected — security risk; shared across all jobs on the runner).

## R-004: Apple notarization with notarytool

**Decision**: Use `xcrun notarytool submit` with an App Store Connect API key (preferred
over App-Specific Password because API keys do not expire and are not tied to a personal
Apple ID 2FA state).

**Required GitHub Secrets**:
- `NOTARIZATION_APPLE_ID` — Apple ID email (only if using App-Specific Password path)
- `NOTARIZATION_TEAM_ID` — Apple Developer Team ID
- `NOTARIZATION_APP_SPECIFIC_PASSWORD` — App-Specific Password (simpler initial setup)

**Alternative (API key path)**:
- `NOTARIZATION_API_KEY_ID`
- `NOTARIZATION_API_ISSUER_ID`
- `NOTARIZATION_API_KEY_BASE64` — base64-encoded .p8 private key

**Decision for v1**: Use App-Specific Password path for simplicity; document migration
to API key path in CONTRIBUTING.md for future maintainers.

**Notarization steps**:
```bash
# Submit for notarization and wait for completion
xcrun notarytool submit "build/Drawl.dmg" \
  --apple-id "$NOTARIZATION_APPLE_ID" \
  --team-id "$NOTARIZATION_TEAM_ID" \
  --password "$NOTARIZATION_APP_SPECIFIC_PASSWORD" \
  --wait \
  --timeout 30m

# Staple the ticket to the DMG
xcrun stapler staple "build/Drawl.dmg"
```

**Rationale**: `--wait` blocks the CI step until Apple's service responds (pass/fail).
`stapler staple` embeds the notarization ticket into the DMG so Gatekeeper can verify
it without network access at install time.

**Alternatives considered**: `altool` (deprecated by Apple since Xcode 14; rejected).
Fastlane `notarize` action (rejected — same toolchain dependency concern as R-003).

## R-005: DMG signing, notarization, and stapling

**Decision**: The `.app` inside the DMG must be signed. The DMG itself must also be
notarized and stapled. The correct order is:

1. Build the `.app`
2. Sign the `.app` with `codesign` (hardened runtime `--options runtime`)
3. Package the signed `.app` into a DMG using `create_dmg.sh`
4. Notarize the DMG with `xcrun notarytool submit --wait`
5. Staple the notarization ticket to the DMG with `xcrun stapler staple`

**Rationale**: Apple's Gatekeeper checks both the `.app` signature and the DMG's
notarization ticket. If either is missing, users see a Gatekeeper warning. Stapling
enables offline Gatekeeper verification (no phone-home required at install time).

**Modifications to create_dmg.sh**: The existing script handles steps 3 only. The
release workflow will invoke the script after step 2 and before step 4. No modifications
to the script itself are required; signing and notarization steps live in the CI YAML.

## R-006: Preventing secrets from leaking to fork PRs

**Decision**: Use the `pull_request` event (not `pull_request_target`) for CI runs on
external contributor PRs. GitHub's security model ensures secrets are unavailable in
`pull_request` workflows triggered from forks.

**Security model**:
- `pull_request` from a fork: runs with read-only `GITHUB_TOKEN`; no repository secrets.
- `pull_request` from a branch in the same repo: has access to secrets.
- `pull_request_target`: runs in the context of the target repo (has secrets) — **dangerous
  for untrusted code**; must never be used for running contributor code.

**Implementation**: The `ci.yml` workflow uses `on: pull_request` only. The signing
secrets are only in `release.yml`, which triggers on `push: tags: v*.*.*` — a write
operation that only repository maintainers with push access can perform.

**Rationale**: This is GitHub's built-in security isolation model. No additional
configuration is needed beyond using the correct event trigger.

**Alternatives considered**: Environments with required reviewers (adds friction; not
needed since signing only runs on tags, not PRs). `pull_request_target` with explicit
SHA checkout (rejected — complex and error-prone for most maintainers).

## R-007: Changelog generation from git commits

**Decision**: Use `git log` between the previous tag and the current tag, formatted for
GitHub Releases. The `gh release create` command can auto-generate release notes from
merged PR titles and commit messages using GitHub's release notes API.

**Implementation** (using gh CLI auto-generate):
```bash
PREV_TAG=$(git describe --tags --abbrev=0 HEAD^ 2>/dev/null || echo "")
CURR_TAG="${GITHUB_REF_NAME}"  # set by GitHub Actions for tag pushes

if [ -n "$PREV_TAG" ]; then
  NOTES_FLAGS="--generate-notes --notes-start-tag $PREV_TAG"
else
  NOTES_FLAGS="--generate-notes"
fi

gh release create "$CURR_TAG" \
  --title "Drawl $CURR_TAG" \
  $NOTES_FLAGS \
  "build/Drawl.dmg#Drawl.dmg"
```

**Rationale**: `gh release create --generate-notes` uses GitHub's release notes API,
which groups changes by PR labels (Enhancements, Bug Fixes, etc.) and lists merged PRs
since the last release — more useful than raw commit messages. This requires the `gh`
CLI, which is pre-installed on all GitHub-hosted runners.

**Alternatives considered**: `git-cliff` (rejected — additional tool dependency).
Manual release notes (rejected — error-prone and slows release cadence).
Hand-crafted `git log --oneline` (acceptable fallback but less structured).

## R-008: Branch protection configuration

**Decision**: Configure branch protection via the GitHub repository settings UI (maintainer
one-time setup) rather than via `gh api` in CI. Document the required settings in
CONTRIBUTING.md under a "Maintainer Setup" section.

**Required settings for `main` branch**:
- Require a pull request before merging: **enabled**
- Required approving reviews: **1**
- Dismiss stale reviews when new commits are pushed: **enabled**
- Require status checks to pass before merging: **enabled**
  - Required checks: `Build and Test` (from `ci.yml`)
- Require branches to be up to date before merging: **enabled**
- Restrict who can push to matching branches: **only maintainers** (i.e., no direct pushes)

**Rationale**: Branch protection via the UI is a one-time operation that persists in
GitHub's repository settings. Automating this via `gh api` is possible but adds
complexity to setup documentation and requires a personal access token with admin
permissions. For a small project, manual UI setup is simpler and more visible.

**Alternatives considered**: `gh api` automation (deferred — appropriate for larger
organizations with many repos). Rulesets API (newer GitHub feature; more powerful but
requires additional permissions understanding; deferred).

## R-009: Canonical repo structure for an open source macOS app

**Decision**: Adopt a clean, contributor-friendly root layout. Move scripts out of the root
into `scripts/`, remove internal developer tooling from the public tree (or gitignore it),
and add standard open source root files. See the intended final layout below.

**Intended root layout**:

```
drawl/
├── .github/
│   ├── workflows/
│   │   ├── ci.yml
│   │   └── release.yml
│   ├── ISSUE_TEMPLATE/
│   │   ├── bug_report.md
│   │   └── feature_request.md
│   └── PULL_REQUEST_TEMPLATE.md
│
├── Drawl/                      # Swift source (unchanged)
├── DrawlTests/                 # Test suite (unchanged)
├── Drawl.xcodeproj/            # Xcode project (unchanged)
│
├── scripts/
│   └── create_dmg.sh           # Moved from root; update release.yml path accordingly
│
├── specs/                      # Feature specifications (speckit; public — contains constitution)
│
├── CONTRIBUTING.md             # New
├── CODE_OF_CONDUCT.md          # New
├── SECURITY.md                 # New
├── LICENSE                     # New (MIT)
├── README.md                   # New
└── project.yml                 # XcodeGen definition (unchanged)
```

**Files to add to `.gitignore`** (internal tooling, not relevant to contributors):
```
# AI / developer tooling
.claude/
.agents/
.specify/
context/

# Profiling
*.profraw
```

**Rationale**: Standard macOS open source repos (Whisky, LinearMouse, AltTab) follow this
convention: source in a named directory, scripts in `scripts/`, all standard open source
root files present, internal dev tooling gitignored. This makes the repo welcoming to
contributors who don't use the same AI tools as the maintainer, and keeps the GitHub
file browser clean.

**Note on `.specify/`**: The constitution and specs contain valuable project context.
Consider keeping `specs/` public but gitignoring `.specify/` (tooling config), or keeping
both public as a transparency measure. Decision is deferred to the maintainer; either is
valid.

**Alternatives considered**: Keeping `create_dmg.sh` at root (acceptable but clutters root
alongside source code). Committing `.claude/` (leaks AI configuration; not relevant to
contributors). A `Makefile` as single entry point (good pattern for future; deferred to
keep initial scope minimal).
